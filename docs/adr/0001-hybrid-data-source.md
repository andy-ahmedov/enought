# ADR-0001: Hybrid data source for phone-spend detection

## Status
Accepted

## Context

Проекту нужен источник данных о тратах **через смартфон**, с максимальным упором на:

- реализуемость для стороннего Android-приложения;
- законность и совместимость с платформой;
- устойчивость к изменениям чужих приложений;
- понятный путь к MVP.

Варианты, которые были рассмотрены:

- прямое чтение данных из Mir Pay;
- подключение к NFC / HCE / tap-to-pay слою;
- Accessibility-скрейпинг;
- SMS как основной канал;
- root / системные хаки;
- уведомления Mir Pay;
- уведомления банков;
- гибрид Mir Pay + банк + ручная правка.

## Decision

Выбран вариант:

**`NotificationListenerService` + Mir Pay notifications + bank notifications + manual correction`**

Конкретно:

1. `NotificationListenerService` — базовый вход в систему.
2. Mir Pay используется как важный сигнал о том, что транзакция была совершена смартфоном.
3. Банковские уведомления используются как дополнительный сигнал и подстраховка.
4. Система локально объединяет эти сигналы.
5. Спорные кейсы уходят в ручное подтверждение / исправление.

## Consequences

### Positive

- архитектура реалистична для публичного Android-приложения;
- нет зависимости от приватных API;
- система может стартовать быстро;
- можно расширять поддержку банков по мере надобности;
- есть честный fallback на ручную коррекцию.

### Negative

- точность не будет идеальной с первого дня;
- придется писать parser-ы под разные шаблоны уведомлений;
- часть устройств и прошивок будет вести себя по-разному;
- без второго источника некоторые события будут оставаться `low confidence`.

## Rejected alternatives

### Direct Mir Pay integration as core data source
Rejected.
Слишком хрупко и зависит от чужого приватного поведения.

### NFC/HCE payment-layer interception
Rejected.
Это переводит проект в класс собственного кошелька / системной интеграции, а не пользовательского приложения-виджета.

### Accessibility scraping
Rejected.
Плохая продуктовая и платформенная ставка.

### SMS as primary source
Rejected for V1.
Слишком тяжелый policy surface и не соответствует минималистичной архитектуре проекта.

### Root/system hooks
Rejected.
Не подходит для реального пользовательского продукта.

## Notes for future

Если когда-нибудь будет пересматриваться архитектура, то сначала исследуются только такие направления:

- новые **легальные** источники сигнала;
- партнерская интеграция более высокого уровня;
- дополнительные способы верификации событий.

Но до отдельного решения пользователя этот ADR считается базовым направлением проекта.

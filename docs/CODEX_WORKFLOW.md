# CODEX_WORKFLOW.md

Этот файл объясняет, как работать с Codex именно в этом проекте.

Цель простая: **не давать Codexу разъезжаться по scope и слоям**.

## 1. Главный режим работы

Для этого репозитория лучший подход:

- короткая задача;
- короткий план;
- маленький diff;
- проверка;
- обновление docs при необходимости.

Не нужно пытаться закрыть весь проект одним запросом.

## 2. Рекомендуемый паттерн промпта

Хороший базовый шаблон:

> Сначала прочитай `AGENTS.md`, `docs/ARCHITECTURE.md` и relevant ADR. Потом предложи короткий план. После этого реализуй только [узкий кусок]. Не трогай [перечень исключений]. В конце перечисли измененные файлы и как проверить результат.

Это сильно снижает вероятность дрейфа.

## 3. Когда использовать `/plan`

Используй `/plan`, если:

- задача нетривиальная;
- затрагивает больше одного слоя;
- есть архитектурный риск;
- задача пока сформулирована слишком общо.

Хорошие примеры:

- спроектировать storage layer;
- придумать parser contract;
- добавить widget states;
- разложить feature на этапы.

Плохие примеры:

- переименовать класс;
- дописать один `when` в UI;
- добавить одно поле в data class.

## 4. Как дробить задачи

Предпочитай такую гранулярность:

### Хорошо

- «Создай domain model `PaymentEvent` и связанные Room entities, без UI и без parser»
- «Добавь `NotificationListenerService`, но пока только сохраняй raw events»
- «Сделай первый Glance widget только для `Ready` и `NoData` state»
- «Добавь review-screen только для подтверждения low-confidence events»

### Плохо

- «Сделай всю архитектуру уведомлений и виджет полностью»
- «Реализуй весь MVP целиком»
- «Сразу поддержи Mir Pay и все банки»

## 5. Как просить у Codex реализацию

### Правильная формулировка

- что входит в задачу;
- что не входит;
- какие файлы можно ожидать;
- какую проверку нужно выполнить;
- надо ли обновить docs.

### Полезные ограничения в промпте

- «Не меняй архитектуру»
- «Не добавляй новые зависимости без отдельной причины»
- «Сделай минимальный рабочий вертикальный шаг»
- «Сначала объясни, какие файлы ты планируешь тронуть»
- «Сконцентрируйся только на app module»

## 6. Как просить review

После реализации полезно отдельным ходом просить review:

> Проведи review сделанного diff с фокусом на correctness, edge cases, privacy и архитектурный дрейф. Код не меняй, только найди проблемы.

Это часто дает более качественный результат, чем смешивать review и реализацию в одном шаге.

## 7. Когда использовать multi-agent mode

Multi-agent — не стандартный режим, а усилитель для сложных задач.

Используй его, когда нужно распараллелить разные типы мышления.

### Хорошие случаи

- один агент исследует текущую архитектуру;
- один ищет privacy-риски;
- один проверяет тестовые пробелы;
- один делает итоговый review diff.

### Пример запроса

> Spawn one subagent to inspect architecture drift, one to inspect privacy and permission issues, and one to inspect test gaps. Wait for all three and summarize findings. Do not modify files.

### Не использовать для

- небольших кодовых правок;
- локальных refactor-ов;
- первых итераций feature slice.

## 8. Про custom agents в этом проекте

В `.codex/agents/` лежат три роли:

- `android_researcher` — только исследование, без изменений;
- `feature_implementer` — аккуратная реализация небольших slices;
- `reviewer` — read-only review correctness / privacy / tests.

Используй их только тогда, когда это реально помогает.

## 9. Про skills в этом проекте

В `.agents/skills/` лежат skills для повторяемых работ:

- `architecture-guard`
- `notification-pipeline`
- `widget-privacy`
- `vertical-slice`

Они нужны, чтобы Codex автоматически подтягивал узкий контекст и не размазывал его по каждому промпту.

## 10. Как фиксировать новое знание

Если во время работы выяснилось что-то важное и повторяемое:

- ошибка в предположении про архитектуру;
- новый жесткий parser edge case;
- новое правило privacy UX;
- лучший порядок шагов,

не оставляй это только в истории чата.
Обнови один из файлов:

- `AGENTS.md`
- `PLANS.md`
- `docs/ARCHITECTURE.md`
- соответствующий `docs/adr/*`
- соответствующий `SKILL.md`

## 11. Чеклист хорошего запроса к Codex

Перед отправкой запроса полезно проверить:

- задача узкая;
- есть понятная цель;
- понятно, что **не** надо делать;
- указан нужный документ-источник;
- есть способ проверить результат.

## 12. Чеклист хорошего ответа от Codex

Ответ считается хорошим, если:

- не уехал за рамки задачи;
- не нарушил ADR;
- перечислил измененные файлы;
- дал способ проверки;
- обновил docs, если реально поменял договоренности.

## 13. Что делать, если Codex поехал не туда

Не спорить абстрактно.
Сразу дать корректирующий запрос такого типа:

> Остановись. Вернись к `AGENTS.md` и `docs/adr/0001-hybrid-data-source.md`. Ты начал двигаться в сторону [кратко]. Для этого проекта это out of scope. Предложи исправленный узкий план в рамках текущей архитектуры.

## 14. Лучший базовый ритм работы

Для этого проекта самый рабочий ритм такой:

1. `/plan` на маленькую задачу.
2. Реализация только одного куска.
3. Отдельный review-ход.
4. Локальная проверка.
5. При необходимости — обновление docs.

Этот ритм лучше длинных заходов «сделай все сразу».

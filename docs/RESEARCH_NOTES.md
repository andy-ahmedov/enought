# RESEARCH_NOTES.md

Краткие заметки по внешним фактам, на которых держится текущая стратегия проекта.

## 1. Почему Mir Pay не является центром архитектуры

- Mir Pay показывает только последние покупки, совершенные со смартфона через Mir Pay.
- Остальные операции Mir сам отправляет смотреть в приложение банка.
- Это подтверждает правильную продуктовую позицию: Mir Pay — важный сигнал, но не единственный источник истины для пользователя.

## 2. Почему не выбрано "подключение к NFC-платежному слою"

- Для полноценной работы на этом уровне проект превращается в собственный wallet / HCE / партнерскую интеграцию.
- Это уже другой класс продукта, а не легкий behavioral widget.

## 3. Почему выбран NotificationListener

- Это системный, пользовательски-разрешенный путь получения posted notifications.
- Для нашего кейса это лучший баланс между реализуемостью, скоростью MVP и соответствием платформе.

## 4. Почему нужен manual correction

- Уведомления на Android не идеальны.
- Разные банки и оболочки устройств отличаются.
- Без ручного подтверждения проект будет казаться магическим, но ненадежным.

## 5. Почему нужен private widget mode

- Домашний экран — публичная поверхность.
- Финансовые данные чувствительные.
- Виджет должен помогать, а не раскрывать лишние детали окружающим.

## 6. Почему V1 без Accessibility и SMS

- Accessibility — плохая ставка и с точки зрения продукта, и с точки зрения платформенных ограничений.
- SMS как основной канал слишком тяжел по policy surface и не нужен для первого MVP.

## 7. Почему в репозитории много опорных markdown-файлов

Это сделано намеренно.
Для Codex в сложном проекте лучше несколько коротких устойчивых источников истины:

- `AGENTS.md` — короткие жесткие правила
- `README.md` — общая картина
- `ARCHITECTURE.md` — технический источник истины
- `ADR` — зафиксированные решения
- `PLANS.md` — текущий план работ
- `skills` — узкие повторяемые сценарии

Так Codex реже фантазирует и реже забывает принятые решения.

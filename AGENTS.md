# AGENTS.md

## Что это за проект

Этот репозиторий разрабатывает Android-приложение и виджет, которые показывают пользователю, сколько он потратил **через смартфон за день**, чтобы вернуть ощущение денег при бесконтактной оплате телефоном.

Главная ценность проекта:

- заметность трат;
- честность распознавания;
- приватность на устройстве;
- маленькие, надежные, проверяемые шаги.

## Зафиксированная архитектура

До явного решения пользователя архитектура считается такой:

- основной источник данных: `NotificationListenerService`;
- источники сигналов: уведомления Mir Pay + банковские уведомления;
- обработка: локальный parser → normalizer → dedupe → confidence;
- fallback: ручное подтверждение / исправление спорных событий;
- хранение: локально на устройстве;
- отображение: экран дня + приватный виджет.

Перед любым архитектурным изменением прочитай:

1. `docs/adr/0001-hybrid-data-source.md`
2. `docs/adr/0002-local-first-and-privacy.md`
3. `docs/adr/0003-v1-exclusions.md`
4. `docs/ARCHITECTURE.md`

## Что запрещено делать без явного запроса пользователя

Не предлагай и не внедряй самовольно:

- прямую интеграцию с внутренностями Mir Pay;
- reverse engineering Mir Pay;
- NFC-sniffing / пассивное чтение платежного стека;
- превращение приложения в собственный кошелек;
- Accessibility-скрейпинг чужих приложений;
- SMS как основной канал в V1;
- root/Xposed/кастомные системные хаки;
- облачную синхронизацию, аккаунты и серверную аналитику;
- расширение продукта до общего трекера всех расходов.

Если задача толкает в одну из этих сторон, остановись и сначала оформи короткую записку с рисками и альтернативами.

## Как работать в этом репозитории

### 1. Работай маленькими шагами

Предпочитай один вертикальный кусок за раз:

- один экран;
- один use case;
- один parser;
- один storage slice;
- один widget state.

Не делай большие расползающиеся изменения без необходимости.

### 2. Для нетривиальных задач сначала план

Если задача затрагивает несколько файлов, слоев или решений:

- сначала прочитай контекст;
- потом предложи короткий план;
- потом реализуй.

Ориентируйся на `PLANS.md`.

### 3. Всегда держи в голове продуктовый фокус

Проект не про «идеальный financial ledger».
Проект про «сколько я уже потратил телефоном сегодня».

Если решение увеличивает сложность, но не усиливает этот core use case, скорее всего оно лишнее.

### 4. Будь честным с уверенностью данных

Если событие сомнительное, отражай это в модели и UX.
Не симулируй точность, которой нет.

### 5. Приватность выше красивых фич

Если задача затрагивает пользовательские финансовые данные, виджет или экран блокировки:

- не выводи лишние детали по умолчанию;
- продумывай private mode;
- продумывай zero/error/permission states.

## Обязательный порядок чтения контекста

1. `README.md`
2. `docs/ARCHITECTURE.md`

## Как понимать "готово"

Задача считается завершенной, только если:

- diff ограничен разумным объемом;
- нет явного архитектурного дрейфа;
- есть способ проверить результат;
- перечислены измененные файлы;
- если изменились правила или архитектура, обновлены docs.

## Что обновлять вместе с кодом

Обновляй документацию, если изменились:

- структура каталогов;
- доменная модель;
- состояние виджета;
- parser contract;
- confidence logic;
- roadmap или порядок этапов.

После каждого завершенного implementation slice обязательно обнови `PLANS.md`:

- что уже сделано;
- что сейчас в `in_progress`, если работа не завершена;
- какой следующий рекомендуемый шаг;
- если изменился порядок ближайших задач, отрази это явно.

Минимум — обнови один из файлов:

- `PLANS.md`
- `docs/ARCHITECTURE.md`
- нужный `docs/adr/*`
- нужный `docs/PROMPT_RECIPES.md`

## Про multi-agent

Используй subagents если задача распараллеливается.

Подходящие случаи:

- архитектурный обзор;
- отдельный reviewer на diff;
- параллельное исследование рисков;
- разделение на explorer / implementer / reviewer.

Неподходящие случаи:

- маленький refactor;
- один экран;
- одна миграция модели;
- один parser.

Проектные custom agents лежат в `.codex/agents/`.

## Ожидаемый стиль изменений

Предпочитай:

- понятные имена;
- минимальные зависимости;
- явные границы между collection / parsing / domain / widget / ui;
- код, который можно тестировать по слоям;
- явные TODO только там, где это честно необходимо.

Избегай:

- скрытой магии;
- плотной сцепки UI и parsing;
- распознавания уведомлений прямо внутри виджета;
- гигантских универсальных классов;
- преждевременной абстракции ради абстракции.

## Если не уверен

Если выбор неочевиден:

1. выбери более простой и локальный вариант;
2. зафиксируй компромисс в комментарии или docs;
3. не расширяй scope без спроса.

## Скиллы

Когда задача совпадает по смыслу, используй профильный skill из `.agents/skills/`:

- `architecture-guard`
- `notification-pipeline`
- `widget-privacy`
- `vertical-slice`

## Последнее правило

Не пытайся быть слишком умным.
Для этого проекта надежность, честность и фокус важнее эффектных, но хрупких решений.

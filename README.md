# Smart Spend Widget

Android-приложение и виджет для осознанности трат при оплате **смартфоном**, а не очередной универсальный трекер личных финансов.

Главная идея проекта:

- оплата телефоном слишком легкая и психологически «безболезненная»;
- пользователь часто не замечает, сколько уже потратил именно **через смартфон** за день;
- домашний экран — лучший способ вернуть видимость трат без открытия банковского приложения;
- приложение должно мягко тормозить импульсные траты, а не превращаться в бухгалтерию.

## Коротко о продукте

Приложение показывает:

- сколько денег потрачено **сегодня через смартфон**;
- сколько осталось до дневного лимита;
- сколько было оплат;
- последнюю оплату;
- при необходимости — спорные операции, которые надо подтвердить или исправить.

Это не антифрод, не банк, не кошелек и не платежный инструмент.
Это **behavioral product**: инструмент, который делает траты заметными.

---

## Продуктовая формулировка

Правильная формулировка проекта:

> Инструмент осознанности трат через смартфон.
> Mir Pay — лишь один из источников сигнала, а не центр архитектуры.

Неправильная формулировка:

> Виджет, который напрямую получает данные из Mir Pay.

Почему это важно:

1. Привязка к одному приложению слишком хрупкая.
2. Прямой доступ к чужому платежному приложению — плохая техническая и продуктовая ставка.
3. Главная ценность здесь не в интеграции с Mir Pay как таковой, а в том, чтобы сделать трату заметной.

---

## Зафиксированное архитектурное решение

Текущее базовое решение для V1 и V2:

**Hybrid / local-first pipeline**

1. `NotificationListenerService` — основной источник событий.
2. Уведомления **Mir Pay** — маркер того, что оплата была совершена смартфоном.
3. Уведомления **банковских приложений** — второй источник для подстраховки, уточнения суммы и дедупликации.
4. Локальная нормализация событий в единый `PaymentEvent`.
5. Локальная дедупликация и выставление `confidence`.
6. Ручное подтверждение / исправление пропущенных или сомнительных операций.
7. Локальная БД.
8. Приватный виджет на домашнем экране.

### Чего проект **не** делает в V1

- не подключается к внутренностям Mir Pay;
- не подслушивает NFC-слой;
- не становится собственным кошельком;
- не использует Accessibility API для скрейпинга чужих экранов;
- не использует SMS как основной канал;
- не требует root, Xposed, модификации прошивки или системных хакающих трюков;
- не отправляет историю трат в облако по умолчанию.

Эти ограничения — не случайность. Это часть продукта.

## Что считать успехом проекта

У пользователя после установки должно появиться ощущение:

- «я вижу, сколько уже потратил телефоном сегодня»;
- «я понимаю, сколько мне осталось до лимита»;
- «я меньше улетаю в незаметные мелкие траты».

Успех проекта — это не максимальная автоматизация любой ценой.
Успех проекта — это **достаточно точное, честное и понятное отображение трат через смартфон**.

## Главные продуктовые принципы

### 1. Осознанность важнее тотального учета
Проект не обязан считать вообще все деньги пользователя.
Он обязан хорошо решать одну задачу: показать траты **через смартфон**.

### 2. Честность важнее магии
Если мы не уверены в событии, мы не делаем вид, что уверены.
Лучше показать пользователю:

- «Похоже, это была оплата телефоном на 349 ₽. Подтвердить?»

чем тихо записать неправильную сумму.

### 3. Приватность по умолчанию
Домашний экран — чувствительное место.
На виджете не должно по умолчанию светиться больше данных, чем нужно.

### 4. Минимальная зависимость от конкретных банков и кошельков
Нужно использовать их уведомления как сигналы, но не завязывать продукт на приватные интеграции.

### 5. Маленькие поставляемые шаги
Проект должен разрабатываться короткими итерациями, чтобы Codex не расползался и не делал огромные неподконтрольные изменения.

## Текущий статус репозитория

На сейчас в репозитории уже есть рабочий bootstrap Android-проекта:

- минимальный `app` module;
- Kotlin + Jetpack Compose;
- базовая навигация и стартовый экран;
- подключенный Room как dependency;
- подготовленная package structure под будущие слои;
- реализованные доменные модели `RawNotificationEvent`, `PaymentCandidate`, `PaymentEvent`, `DailySpendSummary`;
- реализованный raw-event persistence slice на Room: entity, dao, database wiring, mapper и repository skeleton;
- реализованный `NotificationListenerService` skeleton с app-level wiring;
- strict allowlist policy для raw capture из `ru.nspk.mirpay`, `ru.alfabank.mobile.android` и `ru.sberbankmobile`;
- mapper `StatusBarNotification -> RawNotificationEvent`;
- debug-screen сырых уведомлений для проверки ingestion pipeline на устройстве;
- parser contract `NotificationParser` + `NotificationParserRegistry`;
- активный parser только для Mir Pay уведомлений с unit tests;
- raw capture банков сохранен для будущего сбора реальных payment samples, но bank parsing временно выключен в runtime из-за ложных срабатываний на реальных device samples;
- first normalization slice: `PaymentCandidate -> PaymentEvent` с explainable confidence;
- локальный storage для `PaymentEvent` и `payment_event_sources`;
- end-to-end wiring `NotificationListenerService -> parser -> normalizer -> dedupe -> payment event save`;
- `ObserveTodaySummaryUseCase` поверх сохраненных `PaymentEvent` с локальной агрегацией summary за текущий день;
- `TodayScreen`, подключенный к реальному summary flow, со state-моделью `loading / empty / ready / no-permission`;
- read-only список сегодняшних `PaymentEvent` под summary на экране `Today`;
- onboarding / permission flow с отдельным стартовым экраном до `Today`, пока notification access не выдан;
- onboarding / disclosure polish: более честная permission-copy, общий disclosure contract для onboarding и `Today`, и retry-friendly состояние после возврата из системных настроек без выданного доступа;
- первый homescreen widget slice на Glance;
- persisted widget private mode toggle на экране `Today`, который скрывает точную сумму на виджете и при заданном лимите показывает remaining-first private summary;
- локальный дневной лимит с отдельным `Settings` screen, remaining amount в `DailySpendSummary` и мягкими warning thresholds `80% / 100%`;
- manual correction flow для сегодняшних `SUSPECTED` событий: confirm / correct amount / dismiss;
- локальный audit trail ручных правок в `payment_event_edits`;
- первый online dedupe slice для пары `Mir Pay + bank` по сумме и близкому времени, с merged `HYBRID` событием и повышением confidence до `HIGH`;
- duplicate conflict fallback для near-match кейсов: конфликтная пара временно уходит в `SUSPECTED`, исключается из total и разрешается в уже существующем review flow через `merge as one` / `keep both`;
- migration `4 -> 5`, которая автоматически чистит ранее сохраненные standalone `BANK` false positives из локальной БД;
- Gradle wrapper и version catalog;
- локальная проверка через `./gradlew :app:testDebugUnitTest :app:assembleDebug`.

Это значит, что новая сессия Codex не должна снова строить bootstrap с нуля.

Следующий practically useful шаг теперь:

1. собирать реальные raw payment samples банков без включения bank parsing в runtime;
2. после этого возвращать bank parser только для подтвержденных payment templates и уже на них точечно усиливать confidence rules.

Если Android SDK на машине лежит не в `/home/andy/Android/Sdk`, нужно обновить локальный `local.properties` под фактический путь.

## Техническая стратегия кратко

### Основные слои

1. **Collection layer**  
   Слушает posted notifications и складывает сырые события.

2. **Parsing layer**  
   Превращает сырое уведомление в структурированный черновик платежа.

3. **Normalization layer**  
   Приводит сумму, валюту, timestamp и тип события к единому формату.

4. **Matching & dedupe layer**  
   Схлопывает Mir Pay + банк в одно событие, если они относятся к одной оплате.

5. **Persistence layer**  
   Локально хранит сырые уведомления, распознанные события и исправления пользователя.

6. **Widget/UI layer**  
   Показывает пользователю итог за день и дает возможность проверить спорные события.

### Базовые технологии

- Kotlin
- Android SDK
- Jetpack Compose
- Glance для виджета
- Room для локального хранения
- `NotificationListenerService`
- Coroutines / Flow

Допускается уточнение библиотек в процессе, но не смена самой архитектурной идеи без явного согласования.

## Confidence model

Нужна не псевдонаука, а простая и честная логика.

### `HIGH`

- совпали сумма и время из двух источников;
- или есть очень сильный Mir Pay-маркер и сумма распознана однозначно.

### `MEDIUM`

- есть один качественный источник, но без второго подтверждения.

### `LOW`

- сумма похожа на оплату, но не хватает уверенности, что это именно оплата смартфоном;
- или уведомление распознано не полностью.

`LOW` не должен тихо попадать в итог без возможности проверки.

Текущее реализованное правило до dedupe:

- `HIGH` для сильного одиночного Mir Pay сигнала без признаков возврата;
- `MEDIUM` для остальных успешно нормализованных одиночных событий;
- `LOW` при `POSSIBLE_REVERSAL` или `PARTIAL_PARSE`.

## Виджет: что он обязан уметь

Минимум состояний:

1. **Ready**
   - сегодня потрачено;
   - число оплат;
   - лимит / остаток.

2. **Private mode**
   - не показывает точную сумму, только прогресс к лимиту или количество оплат.

3. **Permission missing**
   - объясняет, что нужен доступ к уведомлениям.

4. **No data yet**
   - аккуратный zero state.

5. **Needs review**
   - есть спорные операции.

Виджет должен быть полезным за секунду и не превращаться в маленькое банковское приложение.

## Когда использовать multi-agent mode

Multi-agent нужен не по умолчанию.
Он нужен, когда задача действительно распараллеливается.

Хорошие случаи:

- одна ветка исследует архитектуру текущего состояния;
- вторая ищет тестовые риски;
- третья смотрит UX/permissions edge cases;
- четвертая делает review уже готового diff.

Плохие случаи:

- писать небольшой экран;
- переименовать классы;
- сделать маленький parser;
- добавить один use case.

Для multi-agent в репозитории уже заготовлены custom agents в `.codex/agents/`.
